02 提案陈述

审阅者在 Acme 里留下反馈之后，还得把这些内容重新敲进邮件、Notion 和剪辑室软件——这是流失访谈中被提及最多的一个缺口（41% 的流失团队都点了名）。标注导出可以把任选的一组评论和涂画一键转成 SRT、CSV 或 PDF，用的全是我们已经在运行的基础设施。它已经开发完成、挂在功能开关后面、通过了单会话 1.2 万条标注的压测，而且一个开关就能回滚。从这里到上线，只差下面这四个签核。

03 评审人会问什么

Q1这会不会把仅限内部的评论泄露给客户？
不会。导出走的是和审阅页面同一个可见性解析器：标记为 internal 的标注一律被排除，除非导出者持有 workspace:admin 权限并通过带明确文字说明的复选框主动勾选。已由 14 个权限矩阵测试覆盖。
↗ spec §4.2 — 可见性与权限
Q2碰上超大会话——几千条标注——会怎么样？
导出按每批 500 条标注分页，并以流式方式写出文件。用生产环境里最大的真实会话（一次包含 12,400 条标注的代理商审阅）做压测，3.1 秒完成；常规会话的 p95 是 340ms。
↗ 实现笔记，06-24 — 压测结果
Q3为什么做文件格式，而不是公开 API？
过去两个季度打了 export-request 标签的 212 张支持工单里，SRT + CSV + PDF 能覆盖 83%。API 适合作为第二阶段——它需要这次不涉及的鉴权作用域设计。文件能现在就解除用户的阻塞，又不会堵死 API 的设计空间。
↗ 指标 — 支持工单标签分析，Q1–Q2
Q4这会新增需要我们运维的基础设施吗？
零新增服务。SRT/CSV 在现有的 worker 池（services/render-workers）里内联渲染；PDF 复用已经在生成每周摘要的报表服务。没有新队列，没有新数据存储，没有 schema 迁移。
↗ spec §5.1 — 架构
Q5导出的客户内容，合规上怎么交代？
每一次导出都会在文件元数据里盖上工作区 ID、发起用户和时间戳，并写入审计日志（audit.annotation_export）。文件通过会过期的签名 URL（24 小时）分发，过期后绝不留存。
↗ spec §6 — 审计与留存

05 风险、回滚，以及需要各位做的事

风险与回滚

  • 回滚只需一个开关。把 annotation_export 关掉即可——没有发布任何 schema 迁移，没有需要回退的东西。
  • 影响范围有边界。导出跑在共享 worker 池里，单任务 30 秒超时；失控任务不可能拖垮审阅播放。
  • 最坏的可信故障：签名 URL 清理滞后时会留下孤儿导出文件——但无论如何它们都会在 24 小时后自行过期。
  • 已知缺口：涂画标注在 CSV 里会被压平成包围盒（完整矢量仅 PDF 支持）。产品内已有说明；列为第二阶段事项。
