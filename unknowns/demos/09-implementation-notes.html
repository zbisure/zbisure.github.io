偏离 1 / 414:41

历史遗留批注不一定有帧时间戳

计划怎么说
每条批注都有 frame_ts；烧录渲染器可以把它们全部排序并放上时间线。
代码揭示了什么
annotations 表约 12% 的行早于 migration 0087，其 frame_ts = NULL——在逐帧批注出现之前，它们本是针对整个评审的一般性评论。
保守选择
视频烧录中排除无时间戳的批注；把它们放进 CSV 附属文件，frame_ts 留空并加 legacy_comment 标记。导出包里没有任何内容被悄悄丢弃。
待回头再议
可以用 created_at 相对评审会话的偏移插值出一个时间戳——需要判断这种做法是否足够诚实、适不适合出现在视频里。

偏离 2 / 415:22

队列一来一回，Date 对象变成了字符串

计划怎么说
ExportJobPayload.requestedAt: Date，透传给 worker，并与评审的 updated_at 比较，用来检测过期导出。
代码揭示了什么
BullMQ 会把 payload 以 JSON 序列化后写入 Redis，所以 worker 收到的是 ISO 字符串而非 Date。这个过期比较在单元测试里靠隐式类型转换侥幸通过，却在集成测试的一次重投递任务上失败了。
保守选择
把 payload 按线上传输格式定义类型（requestedAt: string，ISO-8601），在 exportConsumer.ts 的 worker 边界处只解析一次。不写自定义序列化器。
待回头再议
其他 media-tasks 的 payload 很可能潜伏着同样的 bug——给任务 payload 做一个共享的 zod codec，就能根治这一整类问题。

流式 zip 早已存在——放弃了计划中的 archiver 依赖

计划怎么说
引入 archiver，把导出包（视频 + CSV + 缩略图）打成 zip 流写入 S3。
代码揭示了什么
packages/media/src/zipStream.ts 已实现带背压处理的流式 zip——它支撑着批量资产下载功能，自三月起就在生产环境运行。
保守选择
原样复用 zipStream.ts。少一个依赖，导出包还继承了与批量下载相同的、经过实战检验的背压行为。
待回头再议
zipStream 位于 packages/media，如今却被两个功能使用——值得在清理时把它提升到 packages/shared。

偏离 4 / 416:20

访客评审员能查看评审，却不能下载资产

计划怎么说
导出权限 = 评审级 ACL：能打开评审的人就能导出。
代码揭示了什么
工作区级的 guest_reviewer 角色授予评审访问权，但 can_download_assets = false——客户用它让外部干系人参与评论，同时不让媒体外流。若只检查评审级 ACL，访客就能借导出把源视频带出去。
保守选择
对 guest_reviewer 的所有导出格式一律返回 403，与 assetAccess.ts 里已有的资产下载检查保持一致。这是最严格、也最站得住脚的解读。
待回头再议
或许访客仍应能拿到 CSV（只含批注、不含媒体）。这是产品决策，不该由我拍板——已记录在下方。

定夺访客评审员的导出策略

当前行为（来自偏离 4）：访客对任何导出格式都会收到 403。如果你希望访客能导出不含媒体的批注 CSV，改动大约 10 行（在 exportPolicy.ts 里）外加一个测试——但这会改变「不能下载资产」对客户的含义，所以应当由你来定。既不阻塞合并，也不阻塞 QA。

确认导出包的保留期限

计划没有为生成的导出包指定 TTL。我把它们放在 S3 的 exports/ 前缀下，配了 7 天生命周期规则，与分享链接的过期时间一致——足够保守，也没有任何面向客户的承诺超过这个期限。但如果合同或数据保留政策另有规定，只需改 infra/s3.tf 里生命周期规则那一行。

这些发现如何改变第二次尝试

三行文字，粘进下一版计划或 prompt，下一次运行就不用重新发现今天的这些意外。三条要点保留英文原文以便直接粘贴使用，中文大意见下方。

  1. State the legacy-data caveat up front: ~12% of annotations (pre-migration-0087) have null frame_ts. Decide interpolate-vs-sidecar before implementation starts, not at 14:41 mid-build.
  2. Spec queue payloads as wire types: BullMQ round-trips through JSON, so payload interfaces should use ISO strings, never Date — and a shared zod codec for media-tasks payloads would retire the whole bug class.
  3. Inventory existing utilities before adding dependencies: zipStream.ts and useProgressChannel() both existed and both beat the plan's version. Add a “search for prior art” step to the plan template, and settle the guest-reviewer export policy before the next feature touches ACLs.

中文大意（「复制要点」按钮复制的是上方英文原文）

  1. 把遗留数据的坑提前写进计划：约 12% 的批注（早于 migration 0087）frame_ts 为 null。是插值还是放进附属 CSV，应在实现开始前定夺，而不是等构建进行到 14:41 才临场决定。
  2. 把队列 payload 按线上传输类型来定义：BullMQ 经由 JSON 一来一回，payload 接口应使用 ISO 字符串，绝不要用 Date——再给 media-tasks 的 payload 配一个共享 zod codec，就能消灭这一整类 bug。
  3. 加依赖之前先盘点现有工具：zipStream.ts 和 useProgressChannel() 都已存在，而且都比计划里的方案好。给计划模板加一个「搜索既有实现」的步骤，并在下一个涉及 ACL 的功能之前敲定访客评审员的导出策略。
