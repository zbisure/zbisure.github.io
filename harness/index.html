如果你不是模型，
你就是 Harness

Agent = Model + Harness。模型只是 agent 的一个输入；其余一切——prompts、工具、上下文策略、hooks、沙箱、子 agent、反馈与恢复回路——都是 harness。你已有的三页讲了内层（模型自己转的圈）和外层（谁来推这个圈），这一页专攻此前没有一页系统解剖过的中间那层：模型与循环之间的全部工程。

这里把 2026 年中爆发的 “harness engineering” 讨论收成一张地图——这个词怎么来的、harness 由哪五大件组成、Böckeler 的前馈/反馈双控制、长时 agent 怎么搭跨会话的记忆桥、单线程还是多 agent——最后 12 题验证你真的学会了。一条主线贯穿：配置差距，常常大于模型差距。

§1

心智模型：Agent = Model + Harness

先立一句能扛住整页的骨架。裸模型不是 agent——只有当 harness 赋予它状态、工具执行、反馈回路与可强制的约束后，它才成为 agent。

"If you're not the model, you're the harness."

中文大意：一个 agent 系统里，除了模型，剩下的全部都是 harness。模型只是被插进来的一个部件。

— Addy Osmani《Agent Harness Engineering》；LangChain《The Anatomy of an Agent Harness》独立给出同一两层模型

  • 配置差距 > 模型差距。同一个 Opus，换上定制 harness，能把 Terminal-Bench 成绩从 Top30 提升到 Top5——仅改 harness、不换模型。（Addy / LangChain 双证）
  • harness 模式正在收敛。把 Claude Code、Cursor、Codex、Aider、Cline 并排看，它们彼此的相似度，高于各自底层模型的相似度。（Addy）
  • 模型变强不会让 harness 消失，只会让它「上移」（harness moves up）。更强的模型接管低层杂活，工程重心随之抬到更高层。（Addy）
  • mistake-driven design（贯穿全页的方法）：每当 agent 犯错，就花时间工程化一个方案让它永不再犯——每条约束都能追溯到一次具体失败，并固化进 AGENTS.md / pre-commit hook / 一个 review 子 agent。配套口味：hooks 要「静默成功、响亮失败（quiet success, loud failure）」。（Addy）

这一页的全部内容，都是在回答一个问题：模型之外那层工程，到底该搭些什么、怎么搭得让 agent 转得稳。

§2

三层嵌套地图：内层 / 中层 / 外层

最容易混的三个「循环/工程」其实是套在一起的三层。分清谁负责哪层，才知道这一页在讲哪一块。中层高亮的就是本页主场。

外层 · Loop Engineering（循环工程）设计者负责 · 你设计的系统 · 已由 loop-engineering 页详解人设计的、包在 harness 之外、按时/按目标反复触发并喂养 agent 的系统：谁来 prompt、何时触发、状态记在哪。→ 看外层这一页
中层 · Harness（挽具／脚手架）工程师负责 · 你来搭建配置 · ← 本页主场模型之外、循环之外的一切工程供给：上下文策略、工具与执行、验收/评测、运行时反馈、编排与恢复、hooks、沙箱、子 agent。「这个圈能不能转得稳」的全部工程。
内层 · Agent Loop（智能体循环）模型 + 产品自带 · 你不用造它模型每一轮自己跑的 收集上下文 → 行动（调工具）→ 验证 → 重复（Anthropic 概括 gather context → take action → verify work → repeat），即 ReAct 回路。Claude Code、Codex、Amp 都内置这一层。

一句话记边界：内层是「模型自己转的圈」（产品给你的）；外层是「谁来推这个圈、多久推一次」（你设计的系统，已在 loop-engineering 页讲透）；中层 harness 是「这个圈能不能转得稳」的全部工程——这是本页独占的地盘。Böckeler 还把 coding agent 语境下的 harness 再分成内建 harness（产品自带）与你自加的 outer harness（Guides/Sensors），本页主要讲后者。

§3

谱系：一个词的三级跳（2024 → 2026）

“harness engineering” 不是一天冒出来的。先有「agent = LLM harness」的心智，再由 Anthropic 官方系列把各面逐一工程化，最后在 2026 年中被正式命名、解剖，成为行业术语。诚实标注：命名爆发期几篇只有抓取日期，确切发表日未标注，故不硬排先后。

  • 阶段 0 · 地基（2024-12 → 2025 中）——先有「agent = LLM harness」的心智，还没有「harness engineering」这个词。
  • 2024-12-19 · 官方奠基Anthropic《Building Effective Agents》：确立 workflow 与 agent 之分、五种可组合 workflow 模式，以及 ACI（Agent-Computer Interface）与工具设计是关键投资点。「围绕模型做工程」最早的系统论述。
  • 2025 上半年 · 摊开最小形态Thorsten Ball《How to Build an Agent》：一个 agent = 一个 LLM + 一个循环 + 足够的 token，约 300 行 Go。"The Emperor Has No Clothes." 门槛远低于直觉。
  • 2025-04-18 · 配置面成型Boris Cherny《Claude Code Best Practices》：全篇即「如何配置 Claude Code 的 harness」——permissions / hooks / skills / subagents / plugins + 给 Claude 一个它自己能跑的检查。
  • 2025 全年 · 推到极端Geoffrey Huntley 系列把「极简 harness」做绝：stdlib（规则当标准库）→ specs（specs-first）→ Ralph（"Ralph is a Bash loop"）→ subagents（上下文即稀缺 RAM）。用文件系统/git 当记忆，绕过 context rot。
  • 2025-09-30 · 建立心智Simon Willison：编码 agent = 一段充当 LLM harness 的软件；反馈信号（干净通过的测试套件）能指数级放大 agent。
  • 阶段 1 · 官方系列把各面工程化（2025 秋 → 2026 春）
  • 2025-09-11《Writing Effective Tools for Agents》：工具即 ACI 契约，少而精、返回高信号、用 eval 驱动工具改进。
  • 2025-09-29《Effective Context Engineering》：context engineering 是 prompt engineering 的自然演进；上下文是有限且会退化的资源（context rot），模型有「注意力预算」。
  • 2025-11-26《Effective Harnesses for Long-Running Agents》：两段式 harness（initializer + coding agent），跨上下文窗口的记忆桥。
  • 2026-03-24《Harness Design for Long-Running Apps》：generator + evaluator 双 agent；长任务用 context reset 而非 compaction。
  • 阶段 2 · 正式命名与解剖，成词爆发（2026 年中）
  • 2026-06-21 · 中文圈先落地命名dolphin07《SDD 实战手记 08》把 harness 定义为让 Spec 持续生效的三类工程供给：Context（燃油）+ Eval（质检）+ Runtime Feedback（仪表盘）。
  • 2026 年中 · 发表日未标注Addy Osmani《Agent Harness Engineering》定名「Agent = Model + Harness」「配置差距 > 模型差距」「mistake-driven design」；Böckeler《Harness Engineering》给出最严谨框架 Guides/Sensors 双控制；LangChain《The Anatomy of an Agent Harness》做组件解剖；Latent Space 访谈证明「5 个月百万行、几乎 0 行人工手写」的规模化上限。
  • 阶段 3 · 循环工程接棒——harness 之上再叠一层，已由本站 循环工程页覆盖，本页只互链。

一句话谱系：agent = LLM harness（2024–2025 建立心智）→ Anthropic 官方系列把工具/上下文/长时/eval 逐面工程化（2025 秋–2026 春）→ 2026 年中 Addy 命名公式、Böckeler 给框架、LangChain 做解剖，“harness engineering” 成词 → 循环工程在其上接棒。

§4

Harness 五大件解剖

把 LangChain 的组件清单、Addy 的核心组件、Böckeler 的控制分类、Anthropic 各面官方文，收敛成一张部件图——五件。记忆口诀：上下文（知道什么）· 工具（能做什么）· 验收（做对没有）· 运行时反馈（正在发生什么）· 编排恢复（多轮/多手怎么协同兜底）。

件 1 · 上下文供给 Context Provisioning

燃油
定义
agent 开始前/过程中知道的一切系统约束与任务燃料。不是知识库，是本次任务的燃料。
要点
上下文是有限、会退化的资源（context rot），模型有「注意力预算」，目标是能达成结果的最小高信号 token 集；知识要分层 + 保鲜（组织红线 + 项目约束，继承+覆写；每条带出处/owner/失效条件——过期知识比没有知识更危险）；just-in-time 检索优于预塞。真实可用窗口远小于标称：Sonnet 名义 200k，扣系统提示与 harness 后约 176k，MCP 工具易吃掉约 76k。「Less is more」。
出处
Anthropic《Effective Context Engineering》· EngineeringHarness / dolphin07 · Ghuntley《How to Build a Coding Agent》

件 2 · 工具与执行 Tools & Execution / ACI

手
定义
agent 与外部世界交互的契约与执行层——文件系统、bash、编辑、搜索、沙箱。
要点
一个工具 = 四要素（name + 给模型的 description + 输入 JSON schema + 本地执行函数）；五个基础原语 read / list / bash / edit / search(ripgrep)。少而精、别机械包裹 API：把多步合并进单次高层调用减少往返；清晰 namespacing；返回高信号（语义化标识、concise/detailed 两档、分页、可操作的错误提示）。工具太多/功能重叠会稀释注意力，削减常比增加更有效；用 eval 让 Claude 读 transcript 自己重写工具描述。工具还要快——延迟直接拖累回路。
出处
Thorsten Ball · Anthropic《Writing Effective Tools》· Ghuntley · Ronacher

件 3 · 验收 / 评测 Eval / Verification Gate

质检
定义
交付后独立于 agent 的验收闸门。生成者不能批改自己的试卷。
要点
三闸门：① 验收覆盖率（shallow 命名匹配 / deep 语义比对）② Spec 漂移检测（结构层 + 语义层）③ 失败熔断（有限次数 → 暂停 → 回滚）。eval 早启动：从真实失败案例取 20–50 个 test case 起步，别等「完美」；组合 grader = 代码型（确定性断言）+ 模型型（LLM-judge）+ 人评校准；同时测能力（低通过率暴露空间）与回归（≈100% 通过率抓退化）。generator ≠ evaluator：把「做事」与「评判」拆成两个 agent，是长任务可靠性的最强杠杆之一。
出处
EngineeringHarness / dolphin07 · Anthropic《Demystifying Evals》《Harness Design》

件 4 · 运行时反馈 Runtime Feedback

仪表盘
定义
写代码时的即时环境反馈，让 agent 能自诊断、自恢复、闭合回路。
要点
四件旧工程债现在是前提：一键启动环境（make dev）+ 种子数据（测试账号/订单）+ 关键路径冒烟测试（零 flaky）+ 结构化日志（JSON + request id）。以前是加分项，现在是 agent 能否自主工作的前提——旧工程债直接变成 AI 债。反馈信号是放大器（Willison）；构建回路要短——保持 <1 分钟是规模化的关键加速器（Latent Space）；充分日志作为执行副产物，让 agent 自我诊断恢复（Ronacher）。
出处
EngineeringHarness · Simon Willison · Latent Space · Ronacher

件 5 · 编排与恢复 Orchestration & Recovery

调度台
定义
控制流、子 agent 的派生与隔离、失败隔离与回退手段——决定多轮/多 agent 怎么协作、出错怎么收。
要点
子 agent 隔离上下文：主 agent 当调度器，把昂贵的搜索/审计 fan-out 给隔离 context 的 subagent，只回传摘要，防主线程 death spiral。失败隔离 + reinforcement：让 sub-agent 私下迭代、只上报成功；用上下文编辑移除无用失败痕迹但保留教训；在 loop 中不断注入目标提醒/任务状态/失败提示。恢复与 hooks：规划/验证回路 +「静默成功、响亮失败」；共享状态靠（虚拟）文件系统而非复杂 IPC。
出处
Ghuntley《subagents》· Ronacher · Addy · LangChain

对齐中文圈最早的命名：dolphin07 的「三类供给」= 件 1（上下文）+ 件 3（验收）+ 件 4（运行时反馈）。本页在其上补齐件 2 工具与件 5 编排，合成完整五件——这是把散在五六篇里的拆法收成一张图，全站首次。

§5

双控制框架：Guides vs Sensors（Böckeler）

最严谨的 harness 拆法。外层 harness = 两类控制，每类再按「确定性计算 / LLM 推断」分成计算式与推断式 → 四象限。价值：把人类隐性经验外部化为显式、可复用的控制，减少 review toil、少浪费 token。

Feedforward · 事前Guides（前馈控制）开始前给方向，防止不良行为、提高「一次做对」的概率。像给 agent 事先划好的护栏。
Feedback · 事后Sensors（反馈控制）过程中观察产出、在问题到达人眼前就触发自我纠错。像装在流水线上的传感器。

四象限速查

计算式（确定性、快）推断式（LLM 语义判断）
Guides · 前馈（事前）linter / 类型系统 / 模板生成 / codemod / pre-commit 规则 / AGENTS.md·CLAUDE.md 约定 / 权限 allow deny规划阶段的 spec 澄清 / skill·SOP 里的领域指令 / prompt 里的 few-shot 示范
Sensors · 反馈（事后）LSP 诊断 / 测试退出码 / 构建·CI 状态 / 脚本断言 / 静态分析review agent 对抗式复核 / generator+evaluator 的 evaluator / 语义 diff 检查

分布原则 “Keep quality left”：按成本与速度把控制铺到开发生命周期上——快检查放 pre-commit，昂贵的推断式检查放集成之后。从三个维度治理质量：可维护性 / 架构适配度 / 行为。（Böckeler，Thoughtworks）

§6

内层探秘：最小 agent loop 只有 300 行

要知道 harness 该补什么，先看清产品替你内置的内层长什么样。Thorsten Ball 把它摊开：一个 agent = 一个 LLM + 一个循环 + 足够的 token，约 300 行。"The Emperor Has No Clothes."

"An agent is an LLM with access to tools that let it modify things beyond its own context window."

中文大意：所谓 agent，就是一个能调用工具、从而改变自身上下文窗口之外事物的 LLM。工具四要素：name + 给模型的 description + 输入 JSON schema + 本地执行函数。

— Thorsten Ball《How to Build an Agent》（Amp / Sourcegraph）

Pseudocode · 内层循环骨架

一个最小编码 agent loop

tools = register([read_file, list_files, edit_file, bash, search])  # 五原语

conversation = [ system_prompt ]         # 系统提示定义可用工具与期望行为(可数百行)
conversation += [ user_task ]

while True:
    response = model.infer(conversation, tools)     # 1) 发给模型
    conversation += [ response ]
    if response.has_tool_calls():                   # 2) 模型请求调用工具?
        for call in response.tool_calls:
            result = tools[call.name].run(call.args)   # 3) harness 本地执行
            conversation += [ tool_result(result) ]    # 4) 结果回灌进对话
        continue                                       # 5) 带新观察再推理
    if is_done(response):                            # 完成信号/停止条件
        break

关键点 模型自主决定何时调哪个工具；harness 负责提取工具请求、执行、把结果回灌进下一轮。上下文纪律：活动之间清空/隔离 context，防 death spiral。

  • 编码 agent = 一段充当 LLM harness 的软件：用不可见的系统提示（可长达数百行）+ 可调用工具扩展 LLM；反馈信号是放大器。（Simon Willison）
  • “There is no moat”：各家产品的差异被模型能力抹平，真正干活的是模型——所以竞争力落到 harness 的工程质量上。（Ghuntley）
  • 这解释了为什么本页把重心放在中层：内层是产品给你的、且各家趋同；你能拉开差距的，是包在它外面的那层工程。
§7

长时 agent 的 harness：搭一座记忆桥

根本困难：长任务必须在离散会话中工作，每个新会话对之前毫无记忆。仅靠 compaction 不够——前沿模型在循环里会「一口气想做太多」或「看到有进展就宣布完工」。Anthropic 两篇给出两套机制。

E1 · 两段式 harness（Effective Harnesses for Long-Running Agents，2025-11-26）

跑一次 · Session 0Initializer agent首次运行搭好环境：写 feature_checklist.json（每项含 pass/fail 状态）、建立「环境校验仪式」脚本。
每次会话 · Session NCoding agent每个会话做增量推进，并为下一会话留下清晰 artifact：描述性 commit + 进度摘要，强制 clean state。
  • 跨会话持久记忆载体：git commit 历史 + 进度日志 + 功能 checklist。
  • feature-driven 增量 + JSON pass/fail 状态：一次只推进「下一个未完成 feature」，防贪多、防过早收工。
  • 把端到端测试（含浏览器）当核心行为；每次会话以「环境校验仪式」开场，先自检环境是否 ready。

E2 · generator + evaluator 双 agent（Harness Design for Long-Running Apps，2026-03-24）

  • 最强杠杆：把「做事」（generator）与「评判」（evaluator）拆成两个 agent。
  • 长任务用 context reset（干净重启）而非 compaction（压缩累积历史）；sprint 式分解 + agent 之间建立显式「契约」——先定义清楚什么叫 done；基于文件的 artifact handoff 跨会话无损接续。
  • 需要可打分、可迭代的评测标准；随模型变强应持续精简 harness，而非不断加复杂度。

机理旁证（Ghuntley）：Claude 3.7 名义 200k，输出质量在 147k–152k 就开始 clip，tool-call 到 tool-call 开始失败——这正是「为什么要 reset / 隔离而不是硬撑」的物理原因。

§8

编排层的架构决策：单线程 vs 多 Agent

这是件 5 编排里最贵的一个选择。两派各有硬论据与硬数字——不是「多 agent 更酷」，而是一道成本/收益的架构题。（多 agent 数字的批判性对冲另见 正本清源页。）

审慎派 · 单线程优先
  • Cognition《Don't Build Multi-Agents》：并行 subagent 基于彼此看不见的假设行动，冲突决策累积放大、产出脆弱。
  • 原则一：Share context, and share full agent traces, not just individual messages.
  • 原则二：Actions carry implicit decisions, and conflicting decisions carry bad results.
  • 推荐单线程线性 agent；超长任务用专门 LLM 做上下文压缩。「跨 agent 传上下文」2025 仍未解决。
并行派 · 隔离 subagent
  • Anthropic 多 agent 研究系统（orchestrator-worker）：内部评测比单 agent 高 90.2%，但 token 消耗约 15×（单 agent 约 4×）——仅当任务价值撑得起才划算。
  • Geoffrey Huntley（Ralph）：主 context 当调度器，用大量并行 subagent 做审计/搜索，靠隔离 context window 防污染；但构建/测试验证只用 1 个 subagent 防 back pressure。
+90.2%多 agent 内部评测相对单 agent
≈15×token 消耗（单 agent 约 4×）
单线程紧耦合/连续决策任务的默认解

「该不该上多 Agent」判断清单（四问全 Yes 才上）

① breadth-first 可并行？子任务能各自独立探索——不能并行就没意义。
② 子任务弱依赖？不需要连续共享决策；紧耦合 → 退回单线程。
③ 价值撑得起 ≈15× token？成本自己先算清，别为炫技买单。
④ 能共享完整 context+traces？避免「看不见彼此假设」的冲突决策。

共识（双方都同意）：多 agent 适合 breadth-first、可并行探索、子任务弱依赖的场景；不适合紧耦合、需连续决策的任务。而且 Anthropic 官方另一处主张——能用单 agent 别上多 agent，能用 workflow 别上 agent：从最简单方案起步，只在确有收益时加复杂度。任一问 No → 退回单线程线性 agent 或 workflow。

§9

可复制模板：拿去就能用

7 个模板 + 1 个微模板，对应五大件与两大架构决策。功能性内容保留英文/结构原文（是拿来用的，不是拿来读的），说明用中文。从上到下，覆盖 harness 的搭建全链。

件 2 工具 · Tool Definition

工具描述四要素

用在：为 agent 新增/审查一个工具时，逼齐四要素 + 高信号返回。

name: <verb_noun, e.g. search_orders>
description: |
  <一句话说清"什么时候该用它"，给模型看，不是给人看。
   写清副作用、返回什么、何时别用。>
input_schema:            # JSON schema, 每个字段带 description
  query: {type: string, description: "..."}
returns:                 # 高信号: 语义化标识 + concise/detailed 两档 + 分页
  format: concise | detailed
  on_error: "<actionable message: 告诉 agent 下一步怎么办>"

纪律 能合并成一次高层调用就别拆成多次往返；工具总数宁少勿多；错误信息必须可操作。（Anthropic《Writing Tools》）

件 1 上下文 · CLAUDE.md / AGENTS.md

Context 分层 + 知识保鲜

用在：写项目约定或知识库时，做到「分层 + 保鲜」。

# 组织红线（继承，所有项目共享，最高优先）
- <合规 / 安全 / 不可承诺项>

# 项目约束（覆写组织默认）
- 构建/测试命令: <agent 猜不到的>
- 架构决策: <项目特有>

# 知识条目（每条必带三元信息，否则不许进库）
- fact: <内容>
  provenance: <出处链接/来源>
  owner: <谁维护>
  expires_when: <失效条件, 如 "v3 发布后作废">

纪律 注入越多未必越好（注意力稀释）；过期知识比没有知识更危险。（EngineeringHarness）

件 3 验收 · Eval Gate

Eval 三闸门清单

用在：交付前的独立验收门禁（生成者不能批改自己的试卷）。

闸门①  验收覆盖率:  [ ] 需求逐条有对应用例  [ ] shallow 命名匹配  [ ] deep 语义比对
闸门②  Spec 漂移:   [ ] 结构层(接口/字段未擅改)  [ ] 语义层(行为未偏离 spec)
闸门③  失败熔断:    [ ] 有限重试 N=___  [ ] 超限 → 暂停  [ ] 暂停 → 回滚/转人工
起步:  从真实失败 case 取 20–50 条建 eval; grader = 代码型 + 模型型 + 人评校准

出处 EngineeringHarness / dolphin07；Anthropic《Demystifying Evals》。generator 与 evaluator 拆成两个 agent。

件 5 编排 · Böckeler

Guides + Sensors 四象限自检表

用在：搭 harness 时自查「前馈/反馈 × 计算式/推断式」四格是否都有覆盖。

              计算式(确定性快)            推断式(LLM 语义)
Guides 前馈   [ ] linter/类型/模板        [ ] spec 澄清 / skill 指令 / few-shot
             [ ] 权限 allow/deny         [ ]
Sensors 反馈  [ ] 测试退出码/CI/LSP       [ ] review agent 对抗复核 / evaluator
             [ ] 静态分析                [ ]
分布: 快检查放 pre-commit, 昂贵推断式放集成之后 (Keep quality left)

出处 Birgitta Böckeler《Harness Engineering》。四格都空的那一列/行，就是你 harness 的盲区。

内层 · Thorsten Ball

最小 agent loop 骨架

用在：想理解/自建内层循环时的起点。五工具原语 read/list/edit/bash/search + while 回灌循环，完整骨架见 §6。约 300 行即可跑起来。

while True:
    response = model.infer(conversation, tools)
    if response.has_tool_calls():
        for call in response.tool_calls:
            conversation += [ tool_result(tools[call.name].run(call.args)) ]
        continue
    if is_done(response): break

出处 Thorsten Ball / Ghuntley。模型决定调什么工具，harness 执行并回灌——这就是 agent 的全部魔法。

件 3+5 长时 · Anthropic

长时 agent 两段式骨架

用在：任务跨多个上下文窗口时。initializer 只跑一次，generator 每会话推进，evaluator 独立打分。

# 会话 0: initializer agent —— 只跑一次
- 搭好环境, 写 feature_checklist.json (每项含 pass/fail)
- 建立"环境校验仪式"脚本 (verify_env.sh)

# 会话 N: generator agent —— 每次会话
1. 先跑 verify_env.sh (环境校验仪式)
2. 读 git log + progress.md + feature_checklist.json 恢复记忆
3. 只推进"下一个未完成 feature" (别一口气想做太多)
4. 端到端测试(含浏览器) 必须过
5. 描述性 commit + 更新 progress.md (clean state) 交下一会话

# evaluator agent —— 独立于 generator
- 按显式"契约(done 的定义)"打分; 不达标打回
- 长任务用 context reset 而非 compaction

出处 Anthropic《Effective Harnesses…》《Harness Design…》。记忆桥 = git 历史 + 进度日志 + checklist。

件 5 编排 · 架构选型

单 / 多 Agent 选型四格

用在：决定该不该上多 agent。四问全 Yes 才上，任一 No 退回单线程。

[ ] breadth-first 可并行?
[ ] 子任务弱依赖 / 无需连续共享决策?
[ ] 价值撑得起 ≈15× token?
[ ] 能共享完整 context+traces, 避免看不见彼此假设?
→ 全 Yes: orchestrator-worker 多 agent (Anthropic: +90.2% / ~15× token)
→ 任一 No: 单线程线性 agent 或 workflow (Cognition: 单线程优先)

出处 Cognition / Anthropic。默认单线程，多 agent 是需要论证才配得上的例外。

微模板 · mistake-driven design（Addy）：每次 agent 犯错 → 追问根因 → 固化为 { AGENTS.md 约束 | pre-commit hook | 一个 review subagent } → 让同类错误永不复发。你的 harness 就是一部「踩坑史」的物化。
